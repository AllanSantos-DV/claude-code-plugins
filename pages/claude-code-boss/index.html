O que entrega

Brain KB

Base de conhecimento local

Busca semântica sobre SQLite e embeddings. Indexa o que importa e recupera contexto antes de cada ação, com rerank por relevância, recência e frequência. É o backend padrão e segue inalterado: nem o recall externo nem o onboarding da v2.2.0 tocam nele.

Backend opcional

Servidor MCP Memory externo

Troque o SQLite local por um servidor de memória seu ou da equipe. A ativação é por usuário, sobrevive ao auto-update e não afeta quem não usa. No backend externo ele fala o contrato compose/ingest e exige memory-server ≥ 2.18 (o pool-warming da v2.1.0 pede ≥ 2.20).

Identidade de projeto

Recall que casa entre máquinas

O escopo vem de um id estável: CCB_PROJECT_ID, o arquivo .claude-boss-project, senão o nome da pasta. Sem id no backend externo, um advisory consent-first te oferece fixá-lo. A memória segue o projeto, não o caminho, e não vaza pra outro projeto.

Aprendizado no loop

Lições sem reler o transcript

A tool capture_lesson destila lições no próprio turno, mescla duplicatas e promove padrões recorrentes a skills.

Execução curada

Sinal acima de ruído

Redireciona comandos ruidosos e injeta contexto de forma advisory, com backpressure para não inflar a janela.

Dashboard

Inspeção sob demanda

Um painel em localhost para ver a KB, os hooks e os logs. Abre quando você chama, não no boot.

Novo na v2.4.0 · portão de integridade

O auto-updater do boss baixava a release e rodava npm install (e postinstall é código arbitrário), validando só que o package.json existia. A v2.4.0 fecha essa porta com um portão de integridade: ele baixa o ZIP e um digest sha256 irmão, confere o hash antes de extrair ou instalar, e aborta se não bater. É o Sprint 2 do review completo, cada correção failing-first, com prova de mutação num gate de 3 avaliadores + re-gate.

Antes: baixar e confiar O updater extraía e passava pro npm install o que quer que a release entregasse, checando apenas a presença do package.json. Um ZIP corrompido em trânsito, ou entregue por um Location de redirect adulterado, ia direto pro postinstall. A v2.4.0 não confia mais no download: verifica antes de tocar.
integrity gate self-update · fail-closed por padrão
  1. GETrelease.ziphttps · redirect só pra hosts do GitHub
  2. GETrelease.zip.sha256asset irmão · o digest esperado
  3. HASHsha256(zip)computado antes de extrair ou instalar

sha256(zip) =?= digest

bate extrai + npm install o install roda só depois do gate passar
diverge · ou digest ausente ABORTA · fail-closed nenhum byte extraído · escape p/ legado: allowUnsignedLegacy
integridade, não autenticidade: o digest é asset irmão da mesma release, então cobre corrupção, MITM e redirect em trânsito, não uma release comprometida (que publicaria um ZIP malicioso com um digest que confere). Fechar isso pede assinatura destacada contra chave pinada, registrada como follow-up e fora deste release.

SHA-256, fail-closed

O digest confere antes de extrair ou instalar. Divergência ou digest ausente aborta por padrão: o caller de produção não passa opção, então o gate é obrigatório. O escape é explícito e só pra release legado: allowUnsignedLegacy.

Só GitHub, só https

Allowlist de host e redirect: o download só segue https pra hosts do GitHub (github.com, *.githubusercontent.com). Um Location apontando pra host de atacante é recusado antes de escrever um byte.

Backups não acumulam

A poda mantém os 3 installed_plugins.json.bak.<ts> mais recentes, por timestamp numérico. O registry para de crescer sem teto a cada update.

Provado antes de existir

performUpdate ganhou seams injetáveis (opts.io) e testes herméticos que provam o gate rodando antes de extrair. Cada fix entrou failing-first, com prova de mutação, num gate de 3 avaliadores + re-gate.

Portão de integridade, não assinatura: cobre o trânsito, não a autenticidade da release. A assinatura destacada fica como follow-up, junto do mesmo padrão no download do JAR do mcp-memory. 413 unitários + 65 suítes de hooks no verde; eslint + version-sync ok.

Novo na v2.3.0 · blindagem de segurança

A v2.3.0 é um release de blindagem: o Sprint 1 do review completo do plugin. Um review adversarial de 5 áreas, mais um gate de 3 avaliadores independentes (reviewer, tester e revisor externo), achou e fechou 7 classes de vulnerabilidade. Cada correção entrou failing-first, com prova de mutação, e sem breaking de API: o comportamento observável só muda para entrada maliciosa, que agora é rejeitada.

Sprint 1 do review completo A superfície mais exposta era a HTTP do dashboard: valor vindo do atacante chegava cru até um execSync de shell, até o HTML e até o path.join. O review tratou toda entrada não-confiável como hostil por padrão: argv no lugar de string de shell, escape no lugar de interpolação, rejeitar no lugar de coagir.
hardening review adversarial · 5 áreas · gate de 3 avaliadores
7 classes de vulnerabilidade fechadas. Cada uma achada por review adversarial, corrigida failing-first e provada por mutação.
  • Command injection dashboard · brain-promote
    -execSync(string de shell)
    +execFileSync(node, [script, ...argv])
  • Stored XSS dashboard · KB · projeto · doctor · modelos
    -onclick="…" com valor cru interpolado
    +escapeHtml · data-* + listener delegado
  • Shell injection embedder ollama
    -shell ollama run …
    +execFileSync(ollama, [run, model], {input}) · stdin
  • Path traversal project id · brain-server · promote · HTTP
    -path.join(brainDir, id) com id cru
    +sanitizeProjectId rejeita / \ : ..
  • DNS-rebinding + CSRF model-router · /metrics/reset
    -POST /metrics/reset sem checagem de origem
    +Host loopback e Origin ausente/loopback
  • Traversal de estático dashboard · serveStatic
    -prefixo de string, escapa pro diretório-irmão
    +limite por DASHBOARD_DIR + path.sep
  • Interpolação PowerShell model-router-ensure
    -nome de env-var cru no comando
    +allowlist antes de interpolar
cada fix failing-first + prova de mutação · sem breaking de API · o observável só muda para entrada maliciosa, agora rejeitada

Rejeitar, não coagir

O novo sanitizeProjectId rejeita ids com separador, : ou .., em vez de tentar limpá-los. Fecha de uma vez o escape do path.join(brainDir, id) e a colisão silenciosa de escopo (orgA/api vs api), em toda a superfície HTTP do dashboard, no brain-server e no brain-promote.

Argv, nunca string de shell

Injeção de comando e de shell morre na fonte: execFileSync com array de argumentos, texto por stdin, sem shell. Vale pro brain-promote do dashboard e pro embedder ollama. Metacaractere de shell não é mais interpretado.

Nada de atacante entra cru no HTML

Todo campo influenciável passa por escapeHtml: título, resumo, tipo e escopo da KB; projeto e modelo; saída do doctor; err.message de log. Os botões de ação trocaram onclick interpolado por data-* e listener delegado. Fim do stored XSS no dashboard.

Só loopback, só allowlist

O POST /metrics/reset do model-router agora exige Host e Origin loopback, o que barra DNS-rebinding e CSRF drive-by. O serveStatic delimita por path.sep (fim do diretório-irmão), e o nome de env-var no PowerShell passa por allowlist antes de entrar no comando.

Provado antes de existir: cada correção entrou failing-first, com prova de mutação pra garantir que o teste pega a regressão. 405 unitários + 65 suítes de hooks no verde; eslint + version-sync ok. Sem breaking de API: só entrada maliciosa muda de comportamento.

Novo na v2.2.0 · onboarding de identidade

Recall que casa entre máquinas depende de um id estável de projeto. Sem ele, no backend externo o escopo cai no basename(cwd): o nome da pasta, que muda de máquina pra máquina e pode colidir no daemon compartilhado. A v2.2.0 fecha esse gap de usabilidade com um advisory guiado no SessionStart, consent-first: ele te oferece fixar o id, sem nunca escrever por você.

O gap que faltava O marcador que fixa a identidade, o .claude-boss-project, já existia. Só que nada lembrava você de criá-lo, então quase nenhum projeto ganhava a identidade estável pra qual o marcador foi feito. No backend externo, o recall escorregava (silencioso) pro basename(cwd), que muda entre máquinas e clones e pode colidir no daemon compartilhado.
SessionStart · project-identity-advisory só no backend mcp-memory

tem id estável por alguma fonte que o handshake usa?

  • CCB_PROJECT_ID · variável de ambiente da sessão
  • .claude-boss-project · marcador em cwd ou ancestral
  • backend.mcpMemory.projectId · id fixado na config

com todas ausentes, o recall cai no basename(cwd): frágil e colidível no daemon compartilhado

injeta 1 advisorypeça ao agente que ofereça (com o seu ok) fixar o id criando .claude-boss-project
o hook nunca escreve o marcador: identidade é escolha humana, o agente só media. Silencioso quando já há id, no backend local ou sob cooldown por-pasta.

Gating fiel ao handshake

Silencioso quando há id por qualquer fonte que o handshake realmente usa: backend.mcpMemory.projectId, CCB_PROJECT_ID ou o marcador em cwd/ancestral. No backend local, onde o basename é o esperado, ele nem roda.

Consent-first: nunca escreve

O hook injeta o pedido, não o marcador. Quem cria o .claude-boss-project é você, com o seu ok. A identidade é uma escolha humana e o agente só media a oferta.

Cooldown por-pasta, atômico

Um stamp por pasta (hash do cwd), gravado via temp+rename. Sem lost update entre sessões simultâneas: o convite não repete a cada abertura da mesma árvore.

Um knob pra desligar

Não quer o convite? onboarding.projectIdentity: false silencia de vez. Sem mudança no servidor: é 100% cliente e best-effort.

Sem mudança no servidor: o advisory roda 100% no cliente e é best-effort, então degrada em silêncio se algo falhar. Fecha o gap que fazia o recall cross-máquina depender de um marcador que ninguém era lembrado de criar; o README ganhou o quickstart Projeto novo?.

O laço fechado · pool-warming (v2.1.0)

A v2.0.0 fez o recall 100% compose, e isso deixou um laço em aberto. A conversa ingerida é gravada home-scoped e só gradua pro spine se acumular sinal de recall; esse sinal vem do search_memory, que o compose não gera (por design). Sem search no caminho, os docs ingeridos ficavam write-only. A v2.1.0 fecha esse laço.

O laço que faltava O Dreaming promove uma memória home pro spine quando ela acumula sinal, e o sinal vem do search_memory, nunca do compose_recall. Resultado até a v2.0.0: tudo que a sessão ingeria entrava, mas nunca voltava pra ser lido.
pool-warming kb.retrieval.compose.poolWarming · default ON

a cada UserPromptSubmit, dois disparos em paralelo:

o que você vê compose_recall injetado · grounding
o que aquece search_memory {includeHome:true} não-injetado · só sinal
  1. o pool home acumula sinal (recordTopRecall)
  2. o Dreaming (async) promove os úteis → procedural[HOME]
  3. o compose_recall passa a ler, injeta como fato no próximo turno

laço fechado: a injeção que você vê continua 100% compose

best-effort: falha ou timeout do warming nunca quebra o recall · em daemon < 2.20 vira no-op gracioso

Aquecimento em paralelo

Junto do compose_recall injetado, o boss dispara um search_memory {includeHome:true} não-injetado. O includeHome (ADR-017) federa project_id = P OU ausente: a busca do boss, antes estrita por projeto, passa a alcançar o pool home sem vazar entre projetos.

Um knob, ligado por padrão

Em kb.retrieval.compose.poolWarming (default on). Best-effort de verdade: falha ou timeout do warming nunca quebra o recall. A injeção que você vê continua 100% compose.

Degradação graciosa

O warming pede memory-server ≥ 2.20 pra alcançar o pool home. Em daemon anterior vira no-op silencioso: o recall via compose segue intacto, nada trava.

Conversa que gradua

Com o pool aquecido, o Dreaming promove os docs úteis pra procedural[HOME] e o compose passa a injetá-los como fato. A ingestão cooperativa da v2.0.0 deixa de ser write-only.

Tudo atrás de gate e testado: 65 suítes de hooks + 382 unitários no verde, com smoke real contra o daemon 2.20 (o includeHome federa o pool home: 0 estrito → 15 federados).

A base · recall em dois níveis (v2.0.0)

O pool-warming se apoia na entrega da v2.0.0: como o boss lembra. No backend externo, o recall vem em dois níveis. Os fatos relevantes entram com o trecho que casou (grounding); as capacidades entram como ponteiros leves, que o agente abre só quando precisa.

Ingestão cooperativa

No Stop, o transcript acumulado da sessão vai pro ingest_conversation. O servidor descobre o template, destila, tipa, escopa e deduplica por event-id. O cliente só envia; reenviar o acumulado é seguro.

Sem vazamento entre projetos

O brain-backend.init() virou project-aware: em outro projeto refaz o handshake e re-escopa, em vez de reusar o primeiro. Chamada sem projeto nunca sobrescreve o escopo atual.

Falha visível (fail-loud)

O recall-health marca cada recall como ok ou degradado; o brain-health avisa no SessionStart quando o recall vem vazio de forma recorrente: daemon < 2.18 ou fora do ar.

Orçamento de injeção + knobs

Em kb.retrieval.compose: includeHomeSpine, maxInjectChars (teto de texto), timeoutMs (compose lento degrada) e um overlay de metadados genérico.

A v2.0.0 já vinha atrás de gate: o recall degrada fail-loud em daemon < 2.18. Registra o vazio, nunca cai numa busca rasa em silêncio.

Como funciona

No SessionStart, se o backend externo está sem id estável de projeto, o boss injeta um advisory consent-first: o agente te oferece fixar o id criando .claude-boss-project. Ele nunca escreve por você, e cala se o id já existe ou sob cooldown.

Você escreve um prompt. O hook UserPromptSubmit chama o Brain e injeta o bloco [BRAIN]: no backend externo, os fatos vêm com o trecho casado e as capacidades como ponteiros. Em paralelo, um search_memory não-injetado aquece o pool home: sinal pro Dreaming, sem tocar no que você vê.

O agente age com a memória do projeto à mão, escopada por um id estável, sem puxar contexto de outro projeto.

No Stop, o boss destila o aprendizado e cura comandos ruidosos. No backend externo, manda o transcript acumulado pro servidor destilar. Nada relê o transcript à toa nem infla a janela.
